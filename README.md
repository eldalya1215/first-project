[신규] 비용 제약 시 무료 모델 사용 및 품질 유지 방안 (평가기준 13 해결)평가 요구사항: 프로젝트 예산 고갈이나 비용 제약 상황이 발생하여 GPT-5.5 대신 무료 모델(오픈소스 혹은 하위 LLM)을 임베딩해야 할 때, 서비스 품질을 유지하기 위한 프로세스 변경 방안이 있어야 합니다.상용 API 비용 제약으로 인해 하위 무료 모델(예: Llama-3-8B 오픈소스 모델 또는 하위 무료 티어 모델)로 전환할 경우, 다음과 같은 3단계 품질 보완 아키텍처로 프로세스를 변경합니다.[사용자 입력] ➡️ [1단계: RAG 데이터베이스 검색] ➡️ [2단계: Few-shot 템플릿 결합] ➡️ [무료/오픈소스 LLM 추론] ➡️ [3단계: Rule-based 안전 필터 검증] ➡️ [안전한 출력]
RAG (Retrieval-Augmented Generation) 기반 지식 주입: 하위 무료 모델은 매개변수(Parameter) 제약으로 인해 정교한 의학적 지식이 부족할 수 있습니다. 이를 보완하기 위해 '50대 여성 표준 운동 가이드라인' 수집 데이터베이스를 구축하고, 유저 입력이 들어오면 관련 안전 문서를 먼저 검색(Retrieval)하여 프롬프트 컨텍스트에 함께 제공함으로써 무료 모델의 지식 공백을 메웁니다.소형 모델 전용 템플릿(System Prompt Structuring) 최적화: 무료 모델일수록 복잡한 자연어 지시를 놓치기 쉽습니다. 따라서 프롬프트를 명확한 태그 형식(<Persona>, <Constraints>, <Output_Format>)으로 세분화하고, 앞서 정의한 되묻기 흐름이 포함된 Few-shot 예시를 강제 배치하여 모델의 탈옥 및 제약 조건 이탈을 방지합니다.Rule-based 포스트 필터링(Post-Processing) 도입: LLM 출력이 완료된 후, 사용자에게 전달되기 전 단계에서 '스쿼트', '런지', '점프' 등 관절염 환자에게 위험한 키워드가 포함되어 있는지 정규표현식(Regex) 기반 백엔드 코드로 2차 검증합니다. 위험 단어가 검출될 경우 출력을 차단하고 재생성을 요청하는 안전 장치를 프로세스에 추가합니다.5. [신규] 운동/의학 정책 변경 시 환각 방지 운영 전략 (평가기준 14 해결)평가 요구사항: 향후 새로운 스포츠 의학 가이드라인 발표 등 업무 정책 변경 시, LLM의 환각을 지속적으로 방지하기 위한 체계적인 운영 및 업데이트 전략이 필요합니다.시간이 흐름에 따라 변하는 최신 의학 정책을 반영하고 모델의 노후화로 인한 환각을 줄이기 위해 '지속 가능한 프롬프트 운영 수명주기(Lifecycle)' 전략을 수립합니다.정기적 시스템 프롬프트 버전 관리 (Git Tagging): 보건복지부나 스포츠의학회의 가이드라인이 개정될 경우(예: 관절염 환자의 새로운 추천 가동 범위 발표), 시스템 프롬프트 내의 제약 조건 섹션을 즉시 업데이트하고 GitHub Repository에 v2.1, v2.2 형태로 버전을 관리하여 배포합니다.골든 데이터셋(Golden Dataset) 구축 및 회귀 테스트: 정책이 바뀔 때마다 기존에 정상 답변을 하던 영역에서 새로운 환각이 발생하지 않는지 검증하기 위해, 대표적인 50대 여성 고객 프로필 50개로 구성된 '골든 데이터셋'을 구축합니다. 프롬프트 변경 시 이 데이터셋을 일괄 테스트하여 Pass/Fail 율을 자동 모니터링합니다.LLM 로그 모니터링 및 실시간 환각 파이프라인: 실제 운영 환경에서 발생하는 유저 대화 로그 중, 사용자가 "AI가 준 운동을 하다가 통증이 생겼다" 또는 "잘못된 운동인 것 같다"라고 피드백한 로그를 분류 시스템(Classification)을 통해 실시간 수집합니다. 해당 로그는 데이터 버킷에 격리되어 운영자가 매주 분석한 뒤 프롬프트 제약 조건을 촘촘히 보완하는 피드백 루프로 작동합니다.6. [신규] 10턴 이상 장기 대화 시 문맥 유지 프롬프트 구조 개선 방안 (평가기준 15 해결)평가 요구사항: 10턴 이상의 대화가 이어질 때 LLM의 컨텍스트 윈도우(Context Window)가 누적되면서 이전 조건을 잊어버리거나(Lost in the Middle) 문맥이 끊기는 문제를 프롬프트 구조적으로 해결할 방안을 명시해야 합니다.장기 대화 시 최초에 입력된 사용자의 고유 신체 조건(예: 나이, 초기 무릎 관절염 등)을 LLM이 망각하는 문제를 해결하기 위해 프롬프트 구조를 시스템적으로 개선합니다.
### 🔄 장기 문맥 유지를 위한 3대 프롬프트 구조 개선안

1. **상시 요약본(Context Summary Banner) 프롬프트 동적 주입**:
   대화가 5턴 이상 지속될 때마다, 백엔드 시스템에서 이전 대화 내용을 압축 요약하여 시스템 프롬프트 바로 하단에 **[현재 회원 정보 배너]** 형태로 실시간 업데이트하여 주입합니다.
   * *예시 구조:* `<Current_User_Status> 나이: 53세 / 기저질환: 무릎 관절염 / 현재 진도: 주간 루틴 수립 완료 </Current_User_Status>` 이 배너가 대화마다 항상 컨텍스트 최상단에 유지되므로 10턴이 넘어가도 조건이 끊기지 않습니다.

2. **턴마다 제약 조건 리마인드 프로그래밍 (Anchor Prompting)**:
   유저가 새롭게 입력하는 사용자 프롬프트(User Prompt) 앞에 시스템이 자동으로 **"[경고: 당신은 50대 여성 시니어 트레이너이며 환자의 무릎 관절염 조건을 절대 잊지 마십시오]"**라는 앵커(Anchor) 메시지를 숨겨서 함께 전송합니다. 이를 통해 모델이 대화 중간에 딴길로 새는 현상을 방지합니다.

3. **응답 구조의 명시적 세션화**:
   출력 규격에 늘 `### [검토된 회원 건강 상태 리마인드]` 항목을 첫 줄에 출력하도록 강제합니다. 모델이 답변을 생성하기 전에 스스로 "이 환자는 53세 여성이며 관절염이 있다"는 사실을 먼저 텍스트로 쓰게 함으로써(Chain of Thought 효과), 이후 이어지는 운동 조언 문맥에서 환각이 일어나는 확률을 수학적으로 제어합니다.
